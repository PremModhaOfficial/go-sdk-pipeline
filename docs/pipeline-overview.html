1. Problem Statement

Building a production-grade Go SDK client by hand is slow, inconsistent, and error-prone — NFRs, Go idioms, resilience, and observability all depend on reviewer diligence. There is no reproducible, auditable path from "here is what we need" to "here is a tested SDK".

2. What the Pipeline Implements

A deterministic, multi-agent pipeline that:

  • Accepts a 14-section TPRD plus two machine-readable manifests (§Skills-Manifest, §Guardrails-Manifest).
  • Runs 5 phases (Intake → Design → Impl → Testing → Feedback) with an optional Phase 0.5 Analyze for extension/update modes.
  • Uses adversarial devil agents to review every artifact before it advances.
  • Enforces quality via 38 mechanical guardrail scripts (G01–G103).
  • Pauses at 10 human-in-the-loop gates for taste, policy, and safety decisions.
  • Writes to a dedicated branch sdk-pipeline/<run-id> — never main, never pushed.
  • Closes the loop with a feedback phase that improves the pipeline itself across runs.

3. Input & Output

Input

TPRD (14 sections): Request Type, Scope + Non-Goals, Motivation, Functional Requirements, NFRs (numeric gates), Dependencies, Config + API, Observability, Resilience, Security, Testing, Breaking-Change Risk, Rollout, Clarifications.

§Skills-Manifest: skills the TPRD expects the pipeline to consume (validated WARN).

§Guardrails-Manifest: Gxx guardrails that must be enforced (validated BLOCKER).

$SDK_TARGET_DIR: target Go SDK git repo (must be clean).

Request modes: A greenfield, B extension, C incremental update.

→

Output

On branch sdk-pipeline/<run-id>:

  • New/modified Go package(s) with Config struct + New(cfg)
  • Tests: unit ≥90% coverage, integration (testcontainers), benchmarks, fuzz
  • Godoc on every export, Example_* functions, README
  • Ownership markers: [traces-to: TPRD-…], [owned-by:], [constraint:]
  • OTel hooks wired through motadatagosdk/otel

Per-run artifacts in runs/<run-id>/: decision log (JSONL), review reports, coverage/bench/leak/vuln reports, run-summary.md.

Persistent: updated baselines/, evolution/evolution-reports/<run-id>.md.

5. Phases in Detail

Phase 0 — Intake

Lead: sdk-intake-agent. Output: canonical tprd.md, manifest checks, mode detection.

Wave What it does Enforced by
I1 Ingest Read NL or TPRD file; structure into 14 sections —
I2 Skills-Manifest Every declared skill exists at ≥ version G23 WARN
I3 Guardrails-Manifest Every declared Gxx has an executable script G24 BLOCKER
I4 Clarifications Max 5 Qs (target 0 with detailed TPRD) G22 INFO
I5 Mode detect A / B / C —
I6 Completeness All 14 sections non-empty; Non-Goals ≥3 G20, G21
I7 HITL H1 User approves TPRD + manifest verdicts human

Phase 0.5 — Extension Analyze (Mode B/C only)

Agents: sdk-existing-api-analyzer, sdk-marker-scanner.

Snapshots existing current-api.json, test-baseline.json, bench-baseline.txt, and writes ownership-map.json (which symbols are MANUAL vs pipeline-owned). This is what makes incremental updates safe.

Phase 1 — Design

Lead: sdk-design-lead. Devils (7): design, dep-vet, semver, convention, security, over-engineering critic, breaking-change (B/C), API-ergonomics.

Produces: api.go.stub (must compile), interfaces.md, algorithms.md, concurrency.md, dependencies.md.

Gated by: G30 (stub compiles), G31 (deps documented), G32/G33 (govulncheck + osv-scanner), G34 (license allowlist), G38 (no tenancy leak).

Phase 2 — Implementation

Lead: sdk-impl-lead. Branch: sdk-pipeline/<run-id>.

Devils (5): code-reviewer, API-ergonomics, over-engineering critic, marker-hygiene, constraint-devil.

Gated by: G40 (no TODOs), G41 (build), G42 (vet), G43 (fmt), G48 (no init), G95–G103 (marker protocol).

Phase 3 — Testing

Lead: sdk-testing-lead.

Wave Check Gate
T1 Unit coverage ≥ 90% per new package G61
T2 Integration via testcontainers (real backends) G66
T3 Flake hunt: -count=3 must be green G63
T4-T5 Benchmarks + benchstat vs baseline G65 (+5% hot, +10% shared)
T6 goleak + -race -count=5 G62
T7 Fuzz (if TPRD §11 lists targets) —
T8 govulncheck + osv-scanner G32 G33
T9 Observability spans/metrics emit —
T10 Mutation score (optional) —
T11 HITL H8 if perf regression human

On failure: leak → back to Impl; vuln → back to Design; regression → H8 gate; flaky → fix in place.

Phase 4 — Feedback & Self-Learning

Lead: learning-engine.

Wave Agent Produces
F1 metrics-collector per-agent quality scores, phase metrics
F2 phase-retrospector pattern mining per phase
F3 root-cause-tracer HIGH/CRITICAL defects traced to origin
F4 skill-drift-detector + skill-coverage-reporter divergence + unused skills
F5 (retired in v0.3.0) full-pipeline golden-corpus replay was retired per CLAUDE.md rule 28; safety moved to learning-notifications.md + 4 compensating baselines
F6 improvement-planner categorized patches + proposals
F7 learning-engine auto-applies safe patches
F8 baseline-manager raises baselines; resets every 5 runs
