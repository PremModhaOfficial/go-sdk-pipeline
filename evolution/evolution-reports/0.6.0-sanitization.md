# Pipeline v0.6.0 — Sanitization Migration

**Released:** 2026-05-01
**Branch base:** v0.5.0 (frozen, untouched throughout)
**Net change:** language-neutralization of shared-core artifacts; physical separation of language-specific content into `go` / `python` adapter packs; symmetry-paired Go ↔ Python skill extractions.

## Why

A prior in-place sanitization pass under-delivered: shared-core agents and skills retained Go-coupled prose (idioms, tool names, code examples) despite carrying language-neutral labels. The user could not tell whether the LLM's "done" claim was accurate without re-doing the audit themselves.

v0.6.0 redoes the sanitization under a stronger contract:

1. **Physical separation.** v0.5.0 stayed frozen; v0.6.0 grew by dependency-closed batches into a fresh directory tree. A file's location IS the audit — present in v0.6.0 = passed the gate.
2. **Machine-checkable trust layer.** `sanitize-tools/audit.sh` runs `check-no-go-leakage`, `check-no-python-leakage`, `validate-packages.sh`, and per-pack cross-language scans. Reports take the form of script output, not LLM prose.
3. **Strict mode.** When the user identified an over-aggressive deletion mid-migration (the GR-001–GR-024 multi-tenant SaaS guardrails in `guardrail-validation`), the change was bounced back: v0.6 file restored from v0.5, audit re-run.
4. **Per-file extraction-log.** `migration-log.jsonl` records every file's pre/post leakage count, mode of change, and where extracted content went.

## `cross_language_ok` flag

A new YAML-frontmatter / HTML-comment flag that the leakage scripts honor: files declaring `cross_language_ok: true` in their first 20 lines are SKIP'd by the strict scan. Used on:

- 6 shared-core skills with intentional side-by-side Go + Python examples or sibling-skill cross-references
- 22 shared-core agents (orchestrators that legitimately reference per-pack tooling in narrative)
- 5 phase contracts (orchestrate language-specific waves)
- 13 top-level + docs/ markdown files (reference per-pack tooling and multi-tenant SaaS context)

The flag is not a license to ignore the gate — it's a declared cross-language scope marker. The scripts log every skip in their output.

## Two latent v0.5.0 bugs fixed

While building the trust layer, two `validate-packages.sh` bugs surfaced under the empty-stub state of v0.6's first batches:

- **F-006:** `set -u` triggered "unbound variable" on associative-array dereferences when manifests were empty stubs.
- **F-007:** `ls glob | wc -l` with `nullglob` set fell through to listing CWD when the glob matched nothing (in v0.5 every dir had files so it never surfaced).

Both fixed in v0.6.0 `scripts/validate-packages.sh`. v0.5.0 is frozen; not back-ported.

## Multi-tenant SaaS context preserved

CLAUDE.md rule 5 ("No multi-tenancy — SDK is a library, tenant context is caller-supplied") is correct **about the SDK itself**: the SDK doesn't bake tenant logic in. But the SDK is built **for** consumer services that operate in a multi-tenant SaaS architecture, and the GR-001–GR-024 guardrails in `guardrail-validation` check that those consumer services respect the platform's multi-tenant invariants (TenantID propagation, schema-per-tenant isolation, JetStream-only inter-service comm, MsgPack wire format, FK bans, JOIN bans, optimistic locking, soft-delete-only, security headers, etc.).

This is a different scope from the G-numbered SDK-pipeline guardrails. The two scopes are now explicitly documented in `guardrail-validation/SKILL.md` via a transition section. F-008 in `migration-findings.md` records the mid-migration correction.

## Migration-log shape

Each entry records:
- `ts`, `batch`, `src`, `dest`, `mode` (verbatim_copy / scrub / scrub_lite / scrub_heavy / create / restore_after_user_correction / bulk_cross_language_ok_flag)
- `extracted` (where Go content went, where Python content went)
- `net_line_delta`, `eyeball_required`, `version_bump`
- `go_leakage_pre`, `go_leakage_post`, `py_leakage_post`, `gate`
- `note` for any clarification

Re-runnable from machine-readable form. No LLM prose required to reconstruct the migration.

## Rollback

If anything in v0.6.0 misbehaves: v0.5.0 is byte-identical to its pre-migration state. Switch the `pipeline_version` consumer back to v0.5.0; the v0.6.0 directory can be deleted without data loss (per-run state is in `runs/<run-id>/` and never touched).
